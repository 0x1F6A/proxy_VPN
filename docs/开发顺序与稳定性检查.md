# proxy_VPN 项目开发顺序与稳定性检查计划

> 目的：把"整体架构 + API 设计"落地为一条**可执行、可验证、可回滚**的实施路线。
> 原则：**由内向外、由静到动、先骨架后血肉、每阶段必须可演示 + 可回归**。
> 角色：1-3 人小团队（后端 / 全栈 / 运维）即可启动 MVP。

---

## 0. 开发总原则

| 原则 | 说明 |
|------|------|
| **单一职责优先** | 一次只交付一个能"独立验证"的能力 |
| **接口先行** | 每个模块先定 API + 数据库 schema，再写实现 |
| **每阶段闭环** | 每个 Phase 结束 = 可演示 + 自动化测试通过 + 文档更新 |
| **可回滚** | 所有变更走 Git PR，所有部署走容器镜像 tag |
| **可观测** | 每个新服务上线必带：日志 + Metrics + 健康检查 |
| **安全左移** | 密钥不入库；CI 加 gosec / trivy 扫描 |
| **小步快跑** | 每个 Phase 不超过 2 周；阻塞 > 3 天必须重新拆分 |

## 13. 跨阶段持续要求

### 13.1 每个 Phase 必做
1. **更新文档**：`docs/api.md`、`docs/architecture.md`、`README.md`
2. **更新测试**：新功能必须带单测 + 至少 1 个集成测试
3. **更新监控**：新接口/任务必须有 Metrics + 告警规则
4. **更新数据库迁移**：所有 schema 变更走 migration（不允许手改线上库）
5. **更新 CHANGELOG.md**

### 13.2 代码评审 (Code Review) 标准
- 至少 1 人 approve
- CI 全绿
- 无 TODO 留到下个阶段
- 涉及钱 / 鉴权 / 密钥 → 必须 2 人 review
- 重构 PR 与功能 PR 分离

### 13.3 分支策略
- `main`：永远可发布，受保护
- `develop`：集成分支
- `feature/<phase>-<name>`：功能分支
- `hotfix/<id>`：紧急修复
- PR 命名：`[Phase4] feat(billing): add alipay native pay`

### 13.4 版本号
- 遵循 SemVer：`v<major>.<minor>.<patch>`
- 每个 Phase 结束发布一个 `v0.x.0` 预发布
- Phase 10 上线发布 `v1.0.0`

## 14. 风险登记与缓解

| 风险 | 缓解措施 |
|------|---------|
| 支付通道审核被拒 | 主体海外注册；USDT 作为备份通道；同通道多账户分摊 |
| 节点被 GFW 封锁 | 多协议（Reality / Hysteria2 / XHTTP）+ CDN 中转 + 域名池 |
| 单点宕机 | 控制面多副本；DB 主从；Cloudflare 故障转移 |
| 节点 token 泄露 | 节点 token 可单独吊销；mTLS 客户端证书定期轮换 |
| 流量统计错乱 | 双写校验（Redis + MySQL），每日定时核对 |
| 法律风险 | 主体 + ToS 明示禁用条款；不留访问日志 |
| 用户数据泄露 | 全字段加密备份；事故响应预案 + 72h 通知机制 |
